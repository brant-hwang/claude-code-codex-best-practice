Intro

왜 이 가이드를 만들었나

Garry Tan(Y-Combinator CEO)이 공개한 gstack을 써보니, 내가 일하는 스타일과 딱 맞는 하네스였다. 기획부터 리뷰, QA, 배포, 회고까지. 마치 Y-Combinator 시절 빠르게 움직이는 초기 스타트업 같은 느낌을 AI 하네스로 구현한 느낌이었다.

그래서 이참에 AI 코딩 환경을 완전히 처음부터 다시 구축하기로 했다. bkit, superpower, oh my codex, oh my claude code 등 수많은 하네스들을 테스트하다 보니 로컬 환경에는 쓸데없는 파일들도 많았고, 정리를 한번 해야겠다는 생각이 들었다.

그리고 다시 하는 김에, 내가 한번 정리해두면, 처음 하는 누군가에게는 도움이 될 것 같아서 이 문서를 작성했다.

Section 03

하네스 엔지니어링이
중요한 이유

3-1

엔지니어링의 진화

시대핵심 역량핵심 질문
2024프롬프트 엔지니어링AI한테 "뭘 해줘"라고 잘 말하는 기술
2025컨텍스트 엔지니어링AI한테 "필요한 정보"를 잘 전달하는 기술
2026하네스 엔지니어링AI가 "제대로 해낼 수밖에 없는 환경"을 설계하는 기술

2024년의 경쟁력: 좋은 프롬프트를 쓰는 사람 → 2025년: 에이전트를 만들 수 있는 사람 → 2026년: AI가 일하는 환경을 설계할 수 있는 사람

3-2

왜 이렇게 변화했나 — LLM의 발전과 함께

  • 초기 LLM (GPT-3~3.5) — 컨텍스트 윈도우가 4K 토큰. 짧은 프롬프트 안에서 최대한 명확하게 지시하는 게 전부. "프롬프트 엔지니어링" 시대
  • 중기 LLM (GPT-4, Claude 3) — 컨텍스트가 128K~200K로 확장. 이제 코드베이스, 문서, 히스토리를 통째로 넣을 수 있게 됨. "뭘 보여줄 것인가"가 핵심이 된 "컨텍스트 엔지니어링" 시대
  • 현재 LLM (Claude 4, GPT-5) — 에이전트가 자율적으로 코드를 쓰고, 커밋하고, PR까지 올림. 이제 문제는 "한 번의 답변"이 아니라 "수백 번의 자율 행동"을 어떻게 제어할 것인가. "하네스 엔지니어링" 시대

Karpathy: "LLM을 CPU, 컨텍스트 윈도우를 RAM이라고 생각하라. 엔지니어의 역할은 OS다 — 작업에 딱 맞는 코드와 데이터를 메모리에 올리는 것."

3-3

① 컨텍스트 엔지니어링 — AI한테 뭘 알려줄 것인가

신입사원한테 아무 설명 없이 "일해"라고 하면 못 합니다. AI도 똑같습니다.

  • 프로젝트 규칙, 코딩 컨벤션, 작업 현황… 이런 정보를 CLAUDE.md, AGENTS.md 같은 문서로 정리해서 넘겨줘야 합니다
  • 에이전트가 읽을 수 없는 정보는 존재하지 않는 것과 같습니다
  • Cognition (Devin 팀): "컨텍스트 엔지니어링은 사실상 AI 에이전트를 만드는 엔지니어의 1순위 업무다"

Anthropic: "장기 실행 에이전트는 이전에 무슨 일이 있었는지 기억 없이 새 세션을 시작한다. 인수인계 문서 없이 교대 근무하는 엔지니어와 같다."

3-4

② 아키텍처 제약 — AI가 하면 안 되는 것을 정하기

볼링장 범퍼처럼, 자유를 주면 삽질하고, 경계를 정해주면 오히려 더 빨리 정답에 도달합니다.

  • 역설적이지만, 제약이 생산성을 높입니다
  • Mitchell Hashimoto (HashiCorp 창업자): "에이전트가 실수할 때마다, 그 실수를 다시는 못 하도록 하네스에 해법을 추가하라"
  • OpenAI Codex 팀: 의존성 흐름, 레이어 분리, 커스텀 린터를 제약으로 걸어 에이전트 신뢰도를 극적으로 향상
  • Karpathy의 CLAUDE.md가 바로 이 접근 — "어떻게 코딩하라"가 아니라 "하지 말아야 할 것"을 명확히 한 것

3-5

③ 가비지 컬렉션 — AI가 만든 쓰레기 치우기

AI는 좋은 패턴도 복제하지만, 나쁜 패턴도 복제합니다. 정리하지 않으면 품질은 시간이 갈수록 떨어집니다.

  • OpenAI 팀은 처음에 매주 금요일을 "AI 슬롭 청소일"로 사용 — 근무 시간의 20%를 정리에 투입
  • 결국 자동 정리 시스템을 하네스에 내장: 백그라운드 에이전트가 아키텍처 위반을 스캔하고 리팩토링 PR을 자동 생성
  • OpenAI: "기술 부채는 고금리 대출이다. 쌓아두면 복리로 불어나고, 조금씩 갚아나가는 게 항상 낫다"

3-6

하네스 엔지니어링의 실제 성과

  • OpenAI Codex — 100만+ 줄 프로덕션 코드, 사람이 직접 작성한 줄 수 0. 5개월간 ~1,500 PR. 엔지니어 3명이 1인당 하루 3.5 PR
  • LangChain — 같은 모델(GPT-5.2-Codex)로 하네스만 개선했더니 벤치마크 52.8% → 66.5%, 랭킹 30위권 밖 → Top 5
  • Stripe — 내부 AI 에이전트가 주당 1,000+ PR을 머지

Martin Fowler: "하네스 엔지니어링은 컨텍스트 엔지니어링, 아키텍처 제약, 가비지 컬렉션을 포함하는 개념이다."

왜 gstack을 하네스로 선택했나

이미 superpowers, bkit, omx 등 다양하게 써봤지만, gstack이 나와 가장 잘 맞는 스타일이라 이번에는 gstack을 메인 하네스로 써보기로 했다. gstack에 대한 상세한 내용은 뒷편에서 더 자세히 다뤄보겠다.

도구강점약점
superpowers다양한 스킬 번들, 커뮤니티 활발opinionated하지 않아 프로세스 강제력 약함, 브라우저 자동화 없음
bkit범용성 높음, 유연한 확장스프린트 워크플로 미지원, QA/배포 파이프라인 부재
gstack기획→리뷰→QA→배포→회고 전 과정 구조화, 100ms급 persistent 브라우저, 병렬 sprint 지원강하게 opinionated, macOS 중심 쿠키 복호화, iframe 미지원

결정 이유: 프로세스 강제력 + 브라우저 자동화 + 품질 게이트를 모두 갖춘 유일한 선택지.

Section 05

gstack
심화

05

5-1

gstack이란?

"AI 코딩 에이전트를 채팅형 도우미가 아니라, 역할이 분리된 가상 개발팀처럼 운영하게 만드는 스킬 팩 + 초고속 브라우저 자동화 도구"

핵심은 새 모델이 아니라 개발 프로세스를 강하게 구조화한다는 점.

GitHub: github.com/garrytan/gstack

5-2

gstack의 본체

  1. SKILL.md 기반 워크플로 스킬들
  2. Bun + Playwright로 만든 persistent browser daemon
  3. 이 둘을 설치/동기화/검증하는 스크립트들

홍보 문구를 걷어내면, 본질은 "프롬프트 운영체제 + 빠른 브라우저 QA"

5-3

왜 쓰나

프로세스 강제

기획→설계→구현→리뷰→QA→배포→회고를 명령형 스킬로 강제

빠른 브라우저

한 번 띄운 Chromium을 유지, 이후 명령 100~200ms

병렬 작업

워크스페이스별 독립 브라우저/상태/로그

품질 게이트

코드뿐 아니라 버그·테스트·문서·릴리즈까지 묶음

5-4

설치 방법

Claude Code

git clone ...gstack.git ~/.claude/skills/gstack cd ~/.claude/skills/gstack && ./setup

Codex

git clone ...gstack.git ~/.codex/skills/gstack cd ~/.codex/skills/gstack && ./setup --host codex
  • ./setup: Bun 확인 → browse 빌드 → Playwright Chromium → 심링크 생성 → 상태 디렉터리 준비
  • 요구사항: Git, Bun, 호스트 에이전트

5-5

실제 사용 흐름

/office-hours→ /plan-ceo-review→ /plan-eng-review→ /plan-design-review
→ 구현 (직접 지시)→ /investigate
→ /qa→ /review→ /ship→ /document-release→ /retro

gstack은 "명령 모음"이 아니라 사실상 sprint 운영 규약이다. "구현 전용" 스킬은 없고, 플랜 리뷰 후 Claude Code에게 직접 구현을 지시한다.

5-6

주요 스킬의 역할

  • /office-hours — 기능 요청을 그대로 받지 않고 "진짜 뭘 만들려는 건지" 기획을 리프레이밍
  • /plan-ceo-review — 제품 방향을 넓히거나 줄이거나 다시 잡을 때
  • /plan-eng-review — 아키텍처, 데이터 플로우, 실패 케이스, 테스트 계획을 굳힐 때
  • /plan-design-review — 구현 전 디자인 완성도를 끌어올릴 때
  • /design-consultation — 디자인 시스템을 처음부터 새로 잡을 때

5-7

구현 후 워크플로 스킬

  • /review — 구조적 버그를 찾는 편집증적 리뷰
  • /qa — 실제 브라우저를 돌려 테스트, 버그 발견 시 자동 수정 → 재검증
  • /ship — PR/테스트/커버리지/릴리즈 마감. 테스트 없으면 프레임워크까지 부트스트랩
  • /document-release — README, CONTRIBUTING 등 문서 드리프트를 잡음
  • /retro — 주간 회고와 팀/개인 메트릭 분석

5-8

브라우저가 왜 중요한가 — /browse

  • Playwright를 감싼 CLI가 아니라 로컬 HTTP daemon에 명령을 보내는 thin client 구조
  • 상태 파일 .gstack/browse.json에 pid, port, token 기록
  • 보안: localhost 바인딩 + bearer token + 0600 권한
  • 요소 선택: snapshot -i로 만든 @e1, @e2 ref (접근성 트리 기반)

5-9

/browse 고급 기능

  • SPA에서 ref 만료 → locator.count()로 감지, "다시 snapshot 떠라" 에러 반환
  • CAPTCHA/MFA → handoff로 visible Chrome, 사용자 통과 후 resume

6-1+

Karpathy의 CLAUDE.md가 왜 화제가 됐나

  • 단 4가지 원칙으로 AI 코딩 에이전트의 고질적 문제(과잉 엔지니어링, 불필요한 리팩토링, 추측성 코드)를 정면으로 겨냥
  • "시니어 개발자가 보면 과하다고 할까? 그러면 단순하게 다시 짜라" — 이 한 문장이 개발자 커뮤니티에서 폭발적으로 공유됨
  • 기존 CLAUDE.md들이 길고 장황했던 것에 비해, 짧고 구체적인 행동 지침만으로 실제 코드 품질이 눈에 띄게 달라짐
  • AI에게 "어떻게 코딩하라"가 아니라 "하지 말아야 할 것"을 명확히 한 역발상이 핵심

6-2

Karpathy의 원칙 1: Think Before Coding

가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러내라.

  • 가정을 명시적으로 밝히기. 불확실하면 물어보기.
  • 해석이 여러 개면 전부 제시 — 조용히 하나만 고르지 않기
  • 더 간단한 방법이 있으면 말하기. 필요하면 반박하기.
  • 불명확하면 멈추고, 뭐가 헷갈리는지 이름 붙이기.

6-3

Karpathy의 원칙 2: Simplicity First

문제를 해결하는 최소한의 코드. 추측성 코드는 없다.

  • 요청 범위를 넘는 기능 없음
  • 단일 사용 코드에 추상화 없음
  • 요청되지 않은 "유연성"이나 "설정 가능성" 없음
  • 불가능한 시나리오에 대한 에러 핸들링 없음
  • 200줄이 50줄로 될 수 있으면 다시 쓰기

6-4

Karpathy의 원칙 3: Surgical Changes

꼭 필요한 것만 건드리고, 자기가 만든 것만 정리한다.

  • 인접 코드, 주석, 포매팅을 "개선"하지 않기
  • 안 망가진 것을 리팩터링하지 않기
  • 기존 스타일에 맞추기 (본인 취향이 달라도)
  • 내 변경으로 생긴 고아만 제거. 기존 dead code는 언급만.

테스트: 모든 변경된 라인이 사용자 요청에 직접 연결되어야 한다.

6-5

Karpathy의 원칙 4: Goal-Driven Execution

성공 기준을 정의하고, 검증될 때까지 반복한다.

  • "유효성 검사 추가" → "잘못된 입력 테스트 작성 후 통과시키기"
  • "버그 수정" → "재현 테스트 작성 후 통과시키기"
  • "X 리팩터링" → "전후 테스트 통과 확인"
1. [단계] → 검증: [체크] 2. [단계] → 검증: [체크] 3. [단계] → 검증: [체크]

A.2

핵심 설계 철학

Command가 흐름을 오케스트레이션하고, Agent가 독립적으로 일하고, Skill이 지식/절차를 재사용하며, Hook이 바깥에서 자동화를 건다.

  • 같이 봐야 하는 루트 파일: CLAUDE.md + .mcp.json
  • .claude만 보면 절반만 보는 셈

A.3

commands/ — 워크플로 컨트롤러

복잡한 작업을 agent 하나에 몰아넣지 않는다:

  1. Command가 흐름을 정의
  2. Agent가 독립 작업을 처리
  3. Skill이 재사용 가능한 절차를 담당
  • 흐름이 눈에 보이고, 사용자 확인 단계를 넣기 쉽다
  • 한 역할이 비대해지지 않고, 재사용이 쉬워진다

A.4

agents/ — 역할 기반 작업자

"fresh isolated context"를 갖는 독립 actor.

필드설명
name / description식별자 / 언제 써야 하는지
allowedTools / model도구 범위 / haiku·sonnet·opus·inherit
maxTurns / skills최대 턴 / preload할 스킬
memory / isolationuser·project·local / worktree

큰 만능 agent 하나보다, 작은 역할 전용 agent 여러 개가 낫다.

A.5

skills/ — 재사용 가능한 지식/절차

A. 직접 호출

command나 사용자에 의해 호출. 현재 컨텍스트에서 동작. 공통 절차 재사용에 좋다.

B. Agent preload

agent의 skills:에 넣어 시작 시 주입. "이 agent가 알아야 하는 작업 지식". user-invocable: false로 메뉴 숨김 가능.

description은 초기 문맥에 들어가고, 전체 내용은 호출 시에만 로드. 긴 설명은 context budget만 잡아먹는다.

A.6

hooks/ — 이벤트 기반 자동화

Agentic loop 안이 아니라 바깥에서 실행.

이벤트시점
PreToolUse / PostToolUse도구 사용 직전/직후
SessionStart / SessionEnd세션 시작/종료
SubagentStart / SubagentStop서브에이전트
TaskCompleted / WorktreeCreate태스크 완료 / 워크트리 생성

이 저장소는 hooks.py 하나로 모으고 설정 파일로 on/off — 배포·공유가 단순.

A.7

rules/ & settings.json

rules/ — 모듈화된 규칙

  • CLAUDE.md에 전부 몰아넣지 않음
  • 특정 파일군에만 적용되는 룰 분리
  • CLAUDE.md를 짧게 유지 가능

settings.json — 전체 제어

  • 권한 allow/ask/deny
  • plansDirectory, 환경변수, 훅
  • MCP auto-approval
  • "편의성과 안전성을 둘 다 구조로 관리"

A.8

agent-memory/ — 동적 지식 축적

모드범위
memory: project팀 공유 자산
memory: user사용자 전역 기억
memory: local개인적, git에 안 들어가는 기억

Skill = 시작 시 주입되는 정적 설계 지식
Memory = 일하면서 축적하는 동적 운영 지식
이 둘을 섞어 쓰는 게 실전 패턴.

A.9

Command vs Agent vs Skill

항목언제 쓰나컨텍스트적합한 역할
Command사용자가 직접 시작현재 세션진입점, 순서 제어
Agent독립 역할 필요새/격리조사, 구현, 리뷰
Skill재사용 지식/절차현재/preload공통 작업 규칙

시작점 → command · 역할 분리 → agent · 재사용 설명서 → skill

A.10

분리 기준 & 아키텍처 패턴

  • CLAUDE.md = 상위 원칙 / rules/ = 좁은 범위 규칙
  • Skill = "처음부터 아는 것" / Memory = "하면서 배운 것"
  • settings.json = Claude 동작 제어 / .mcp.json = 외부 서버 연결

대표 패턴: Command → Agent (with preloaded skill) → Skill
예: /weather-orchestrator → weather-agent(weather-fetcher preload) → weather-svg-creator

A.11

Best Practice (1–3)

  • 1. CLAUDE.md는 짧고 강하게 — 거대한 위키 금지. 200줄 내외. 세부는 rules/로 분리.
  • 2. 범용보다 역할 전용 agent — 권한을 좁히고, memory를 분리하고, prompt drift를 줄인다.
  • 3. Skill은 progressive disclosure — 모든 지식을 항상 주입하지 말고, 필요할 때만 꺼내온다.

A.12

Best Practice (4–6)

  • 4. Hook은 "사후 처리"에 강하다 — PostToolUse로 포맷팅, Stop으로 검증 리마인드.
  • 5. 권한은 wildcard로 관리 — Bash(npm run *), Edit(/docs/**) 패턴 허용. destructive는 ask/deny. "의도적으로 좁게"가 기본.
  • 6. 모노레포는 계층적 설계 — 루트 CLAUDE.md(공통) + 패키지별 CLAUDE.md(도메인). 한 군데에 몰아넣지 않는다.

A.13

Best Practice (7–8)

  • 7. 작업은 작게 쪼개고 context 관리 — plan mode로 시작, 50% 부근에서 /compact, subtask는 50% 이하로 끝낼 것, 자주 커밋. agent/skill/rules를 잘 쪼개는 이유가 결국 context 관리.
  • 8. 사람 승인 지점은 command에 — 배포 직전 확인, destructive action 확인, 옵션 선택. agent에게 전부 맡기면 운영 통제력이 떨어진다.

A.14

실무용 추천 프로젝트 구조

project/ ├── CLAUDE.md # 짧은 상위 규칙 ├── .mcp.json # 외부 서버 연결 └── .claude/ ├── settings.json / settings.local.json ├── rules/ # 도메인별 세부 규칙 ├── commands/ # 사람이 시작하는 워크플로 ├── agents/ # 역할 분리 ├── skills/ # 재사용 지식 └── hooks/ # 자동 포맷/검증/알림

A.15

도입 순서

처음부터 전부 도입하면 과하다. 단계적으로:

CLAUDE.md→ settings.json→ commands→ agents→ skills→ hooks

A.16

가장 실용적인 결론

  1. .claude는 단순 설정 폴더가 아니라 Claude Code 운영 레이어다.
  2. command, agent, skill은 대체재가 아니라 역할이 다른 구성요소다.
  3. 좋은 세팅 = 짧은 메모리 + 모듈화 rules + 재사용 skills + 제한된 agents + event-driven hooks
  4. 팀 공유 자산과 개인 자산을 분리해야 오래 간다.
  5. context budget과 권한 관리를 구조로 해결해야 안정적으로 동작한다.
