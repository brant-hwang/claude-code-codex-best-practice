Intro

Why This Guide Exists

After trying gstack, released by Garry Tan (Y-Combinator CEO), I found it was the perfect harness for my workflow. From planning to review, QA, deployment, and retrospectives — it felt like a fast-moving early-stage startup from the Y-Combinator days, implemented as an AI harness.

So I decided to rebuild my entire AI coding environment from scratch. After testing countless harnesses — bkit, superpower, oh my codex, oh my claude code, and more — my local environment was cluttered with unnecessary files and badly needed a cleanup.

And since I was starting over anyway, I figured documenting the process might help someone doing this for the first time — so I wrote this guide.

Section 03

Why Harness Engineering
Matters

3-1

The Evolution of Engineering

EraCore CompetencyCore Question
2024Prompt EngineeringHow to ask AI the right way
2025Context EngineeringHow to deliver the right information to AI
2026Harness EngineeringHow to design an environment where AI can't fail

2024: people who write good prompts → 2025: people who can build agents → 2026: people who can design the environment where AI works

3-2

Why This Evolution — Driven by LLM Progress

  • Early LLMs (GPT-3~3.5) — 4K token context window. All you could do was craft clear instructions within a short prompt. The "Prompt Engineering" era
  • Mid LLMs (GPT-4, Claude 3) — Context expanded to 128K~200K. Now you could feed entire codebases, docs, and history. "What to show" became the key — the "Context Engineering" era
  • Current LLMs (Claude 4, GPT-5) — Agents autonomously write code, commit, and open PRs. The problem shifted from "one good answer" to "controlling hundreds of autonomous actions" — the "Harness Engineering" era

Karpathy: "Think of an LLM like a CPU, and its context window as the RAM. Your job as an engineer is akin to an OS — load that working memory with just the right code and data for the task."

3-3

① Context Engineering — What to Tell AI

You wouldn't tell a new hire "just work" with zero onboarding. AI is no different.

  • Project rules, coding conventions, work status… deliver this via CLAUDE.md, AGENTS.md
  • Information the agent can't read doesn't exist
  • Cognition (Devin team): "Context engineering is effectively the #1 job of engineers building AI agents"

Anthropic: "Long-running agents begin each new session with no memory of what came before — like engineers working in shifts with no handoff documentation."

3-4

② Architecture Constraints — Define What AI Must NOT Do

Like bowling bumpers — give full freedom and AI flounders; set boundaries and it reaches the answer faster.

  • Paradoxically, constraints boost productivity
  • Mitchell Hashimoto (HashiCorp founder): "Anytime an agent makes a mistake, engineer a solution so it never makes that mistake again"
  • OpenAI Codex team: dependency flow, layer separation, and custom linters as constraints dramatically improved agent reliability
  • Karpathy's CLAUDE.md is exactly this approach — not "how to code" but "what not to do"

3-5

③ Garbage Collection — Clean Up After AI

AI replicates good patterns, but it replicates bad patterns too. Without cleanup, quality degrades over time.

  • The OpenAI team initially spent every Friday as "AI slop cleanup day" — 20% of work time on cleanup
  • Eventually built automated cleanup into the harness: background agents scan for architecture violations and auto-generate refactoring PRs
  • OpenAI: "Technical debt is a high-interest loan. It compounds if left unpaid — it's always better to pay it down continuously"

3-6

Real Results of Harness Engineering

  • OpenAI Codex — 1M+ lines of production code, zero hand-written. ~1,500 PRs over 5 months. 3 engineers averaging 3.5 PRs per person per day
  • LangChain — Same model (GPT-5.2-Codex), only improved the harness: benchmark 52.8% → 66.5%, ranking from outside Top 30 → Top 5
  • Stripe — Internal AI agents merging 1,000+ PRs per week

Martin Fowler: "Harness Engineering includes context engineering, architectural constraints, and garbage collection."

Section 05

gstack
Deep Dive

05

5-1

What is gstack?

"A skill pack + ultra-fast browser automation tool that makes AI coding agents operate like a virtual development team with separated roles, rather than a chatbot assistant."

The key isn't a new model — it's that it strongly structures the development process.

GitHub: github.com/garrytan/gstack

5-2

Core Components of gstack

  1. SKILL.md-based workflow skills
  2. Persistent browser daemon built with Bun + Playwright
  3. Scripts that install, sync, and validate these two

Strip away the marketing, and the essence is "a prompt operating system + fast browser QA"

5-3

Why Use It

Process Enforcement

Forces planning → design → implementation → review → QA → deploy → retro via imperative skills

Fast Browser

Keeps Chromium alive once launched; subsequent commands take 100-200ms

Parallel Work

Independent browser/state/logs per workspace

Quality Gates

Bundles not just code, but bugs, tests, docs, and releases together

5-4

Installation

Claude Code

git clone ...gstack.git ~/.claude/skills/gstack cd ~/.claude/skills/gstack && ./setup

Codex

git clone ...gstack.git ~/.codex/skills/gstack cd ~/.codex/skills/gstack && ./setup --host codex
  • ./setup: Check Bun → build browse → Playwright Chromium → create symlinks → prepare state directory
  • Requirements: Git, Bun, host agent

5-5

Actual Usage Flow

/office-hours→ /plan-ceo-review→ /plan-eng-review→ /plan-design-review
→ Implementation (direct instruction)→ /investigate
→ /qa→ /review→ /ship→ /document-release→ /retro

gstack is not a "collection of commands" — it's essentially a sprint operation protocol. There is no dedicated "implementation" skill; after plan review, you instruct Claude Code to implement directly.

5-6

Key Skill Roles

  • /office-hours — Instead of taking feature requests at face value, reframes "what you're really trying to build"
  • /plan-ceo-review — For broadening, narrowing, or realigning the product direction
  • /plan-eng-review — For locking down architecture, data flow, failure cases, and test plans
  • /plan-design-review — For raising design completeness before implementation
  • /design-consultation — For creating a design system from scratch

5-7

Post-Implementation Workflow Skills

  • /review — Paranoid review that hunts for structural bugs
  • /qa — Runs actual browser tests; auto-fixes bugs on discovery → re-verifies
  • /ship — PR/test/coverage/release wrap-up. Bootstraps test frameworks if none exist
  • /document-release — Catches documentation drift in README, CONTRIBUTING, etc.
  • /retro — Weekly retrospective and team/individual metrics analysis

5-8

Why the Browser Matters — /browse

  • Not a CLI wrapping Playwright, but a thin client architecture that sends commands to a local HTTP daemon
  • State file .gstack/browse.json records pid, port, and token
  • Security: localhost binding + bearer token + 0600 permissions
  • Element selection: snapshot -i creates @e1, @e2 refs (accessibility tree based)

5-9

/browse Advanced Features

  • SPA ref expiry → detected via locator.count(), returns "take a new snapshot" error
  • CAPTCHA/MFA → handoff to visible Chrome, user passes it, then resume
  • "Persistent login, persistent tabs, persistent cookies, fast round-trips" — the key difference from MCP browser tools

5-18

Limitations and Caveats

  • Strongly opinionated — Can feel restrictive if you don't align with Garry Tan's development culture
  • Most of the core logic lives in prompt/skill markdown → more "operating rules" than "engine"
  • Cookie decryption is macOS Keychain-centric (Windows/Linux not implemented)
  • No iframe support (most-requested missing feature)
  • On non-Claude hosts, /careful etc. degrade to advisory prose
  • Telemetry is opt-in but exists — collects skill name, time, result, version, OS

5-19

gstack Conclusion

gstack is not "a tool that makes AI smarter" — it's "a way to run AI like a team."

  • The biggest improvement comes from operational structure, not the model
  • Redefining plans → splitting review/QA/docs/release into skills → browser → parallel sprints
  • Better suited for founders, tech leads, and solo developers who need strong process and role separation than general-purpose agents with high degrees of freedom

Section 06

CLAUDE.md
AGENTS.md Configuration

06

6-1

Recommended CLAUDE.md / AGENTS.md Structure

  • Use Andrej Karpathy's concise CLAUDE.md as the main template
  • 4 core principles: Think Before Coding, Simplicity First, Surgical Changes, Goal-Driven Execution
  • Add Gstack directives
  • Add sub-agent mode directives
  • Add RTK directives (@RTK.md reference)
  • Since RTK currently only works with Claude Code, just remove RTK.md to use the same structure for ~/.codex/AGENTS.md

6-1+

Why Karpathy's CLAUDE.md Went Viral

  • Just 4 principles that directly target AI coding agents' chronic problems: over-engineering, unnecessary refactoring, and speculative code
  • "Would a senior engineer say this is overcomplicated? If yes, simplify." — this single line went viral across developer communities
  • Unlike long, verbose CLAUDE.md files, short and specific behavioral rules alone made a visible difference in code quality
  • The key insight: instead of telling AI "how to code," clearly define "what not to do"

6-2

Karpathy's Principle 1: Think Before Coding

Don't assume. Don't hide confusion. Surface tradeoffs.

  • State assumptions explicitly. If uncertain, ask.
  • If multiple interpretations exist, present them all — don't silently pick one
  • If a simpler approach exists, say so. Push back when warranted.
  • If something is unclear, stop. Name what's confusing.

6-3

Karpathy's Principle 2: Simplicity First

Minimum code that solves the problem. Nothing speculative.

  • No features beyond what was asked
  • No abstractions for single-use code
  • No unrequested "flexibility" or "configurability"
  • No error handling for impossible scenarios
  • If 200 lines can be 50, rewrite it

6-4

Karpathy's Principle 3: Surgical Changes

Touch only what you must. Clean up only your own mess.

  • Don't "improve" adjacent code, comments, or formatting
  • Don't refactor things that aren't broken
  • Match existing style, even if you'd do it differently
  • Only remove orphans your changes created. Mention pre-existing dead code, don't delete it.

The test: Every changed line should trace directly to the user's request.

6-5

Karpathy's Principle 4: Goal-Driven Execution

Define success criteria. Loop until verified.

  • "Add validation" → "Write tests for invalid inputs, then make them pass"
  • "Fix the bug" → "Write a reproduction test, then make it pass"
  • "Refactor X" → "Ensure tests pass before and after"
1. [Step] → verify: [check] 2. [Step] → verify: [check] 3. [Step] → verify: [check]

6-6

Gstack Directive for CLAUDE.md & AGENTS.md

Add the following from the gstack GitHub README.md to the bottom of your CLAUDE.md & AGENTS.md.

markdown # Gstack To use the /browse skill from gstack for all web browsing, never use mcp__claude-in-chrome__* tools, and lists the available skills: /office-hours, /plan-ceo-review, /plan-eng-review, /plan-design-review, /design-consultation, /review, /ship, /browse, /qa, /qa-only, /design-review, /setup-browser-cookies, /retro, /investigate, /document-release, /codex, /careful, /freeze, /guard, /unfreeze, /gstack-upgrade.

Section 07

Hands-on Example

Building a Todo Web App with gstack + Ralph

07

Step 1

Idea Refinement — /office-hours

Claude asks Socratic questions:

  • "Why do you want to build this? What makes it different from existing Todo apps?"
  • "Who is the target user?"
  • "What is the narrowest entry point (wedge)?"

Idea crystallized → "A CLI-first Todo app for developers. The web dashboard is view-only."

Step 2

Strategy Review — /plan-ceo-review

Plan review from a CEO/founder perspective:

  • "Do you really need a dashboard? Isn't CLI alone sufficient?"
  • "Wouldn't it be better to reduce scope and ship the MVP faster?"

Decision: Phase 1 is web CRUD only. CLI moves to Phase 2.

Step 3

Engineering Review — /plan-eng-review

Lock down the architecture:

  • Stack: Next.js 15 (App Router) + Prisma + SQLite
  • Model: Todo { id, title, completed, createdAt, updatedAt }
  • API: /api/todos (GET, POST, PUT, DELETE)
  • Testing: Vitest + Playwright E2E
  • Edge cases: empty title, duplicates, concurrent edits

Step 4

Design System — /design-consultation

  • Generate a minimal design system
  • Color palette, typography, spacing rules
  • Output: DESIGN.md + font/color preview page

Step 5

Implementation — Automated via Ralph Loop

Stepping outside gstack's domain, Ralph enters the picture. With the plan finalized, hand off implementation to a Ralph loop.

/ralph-loop " Phase 1: Project init (Next.js + Prisma) Phase 2: API implementation (/api/todos CRUD) Phase 3: UI implementation (follow DESIGN.md) Phase 4: Testing (Vitest + Playwright, coverage 80%+) " --completion-promise "COMPLETE" --max-iterations 30

Step 5 (continued)

What Ralph Does

  1. Implements Phases 1-4 in order
  2. Runs tests → analyzes failures and fixes them
  3. Re-runs tests → repeats
  4. When all tests pass, outputs COMPLETE → loop terminates

No human intervention needed. Run it overnight and check the results in the morning.

Step 6-7

Code Review & Second Opinion

/review

  • Structural inspection of SQL safety, trust boundary violations, conditional side effects
  • Fix discovered issues

/codex review

  • OpenAI Codex independently reviews the diff
  • Acts as a pass/fail gate

Step 8-9

QA Testing & Design Audit

/qa

  • Click through add/delete/complete Todo in a real browser
  • Catch broken UI, missing error handling
  • Bug → auto-fix → re-verify → commit

/design-review

  • Capture live site screenshots
  • Check spacing, fonts, AI slop patterns
  • Fix visual issues directly in source

Step 10-11

Deployment & Documentation Sync

/ship

  • Merge to base branch
  • Re-run tests (confirm pass)
  • Diff review + VERSION bump
  • CHANGELOG → commit → push → PR

/document-release

  • Verify README matches actual code
  • Update API docs, CONTRIBUTING
  • Clean up TODOs

Step 12

Retrospective — /retro

  • Analyze commits from this sprint
  • Code quality metrics
  • Ralph loop iteration count — identify where the most blockers occurred
  • Derive improvement points for the next sprint

Claude Code
Internals

A

A.1

.claude Is Not Just a Config Folder

DirectoryRole

A.2

Core Design Philosophy

Commands orchestrate the flow, Agents work independently, Skills provide reusable knowledge/procedures, and Hooks attach automation from the outside.

  • Root files to read alongside: CLAUDE.md + .mcp.json
  • Looking at .claude alone only gives you half the picture

A.3

commands/ — Workflow Controllers

Don't dump complex tasks into a single agent:

  1. Commands define the flow
  2. Agents handle independent tasks
  3. Skills provide reusable procedures
  • The flow is visible, and it's easy to insert user confirmation steps
  • No single role gets bloated, and reuse becomes straightforward

A.4

agents/ — Role-Based Workers

Independent actors with "fresh isolated context".

FieldDescription
name / descriptionIdentifier / when to use it
allowedTools / modelTool scope / haiku, sonnet, opus, inherit
maxTurns / skillsMax turns / skills to preload
memory / isolationuser, project, local / worktree

Multiple small role-specific agents beat one large do-everything agent.

A.5

skills/ — Reusable Knowledge/Procedures

A. Direct Invocation

Called by a command or user. Runs in the current context. Great for reusing common procedures.

B. Agent Preload

Added to an agent's skills: field to inject at startup. "Working knowledge this agent needs." Can hide from menus with user-invocable: false.

A.10

Separation Criteria & Architecture Patterns

  • CLAUDE.md = high-level principles / rules/ = narrow-scope rules
  • Skill = "what you know from the start" / Memory = "what you learn while working"
  • settings.json = Claude behavior control / .mcp.json = external server connections

Representative pattern: Command → Agent (with preloaded skill) → Skill
Example: /weather-orchestrator → weather-agent(weather-fetcher preload) → weather-svg-creator

A.11

Best Practice (1-3)

  • 1. Keep CLAUDE.md short and strong — No giant wikis. Around 200 lines. Move details to rules/.
  • 2. Role-specific agents over general-purpose ones — Narrow permissions, separate memory, reduce prompt drift.
  • 3. Skills use progressive disclosure — Don't inject all knowledge at all times; pull it in only when needed.

A.12

Best Practice (4-6)

  • 4. Hooks excel at "post-processing" — PostToolUse for formatting, Stop for verification reminders.
  • 5. Manage permissions with wildcards — Bash(npm run *), Edit(/docs/**) patterns allowed. Destructive actions → ask/deny. "Intentionally narrow" is the default.
  • 6. Monorepos need hierarchical design — Root CLAUDE.md (shared) + per-package CLAUDE.md (domain-specific). Don't dump everything in one place.

A.13

Best Practice (7-8)

  • 7. Break tasks small and manage context — Start with plan mode, /compact around 50%, finish subtasks under 50%, commit often. The reason to split agents/skills/rules well is ultimately context management.
  • 8. Put human approval points in commands — Pre-deployment confirmation, destructive action confirmation, option selection. Delegating everything to agents erodes operational control.

A.14

Recommended Project Structure for Production

project/ ├── CLAUDE.md # Short high-level rules ├── .mcp.json # External server connections └── .claude/ ├── settings.json / settings.local.json ├── rules/ # Domain-specific detailed rules ├── commands/ # Human-initiated workflows ├── agents/ # Role separation ├── skills/ # Reusable knowledge └── hooks/ # Auto format/validate/notify

A.15

Adoption Order

Adopting everything at once is overkill. Go step by step:

CLAUDE.md→ settings.json→ commands→ agents→ skills→ hooks

A.16

The Most Practical Takeaways

  1. .claude is not just a config folder — it's the Claude Code operating layer.
  2. Command, agent, and skill are not interchangeable — they are components with different roles.
  3. Good setup = short memory + modular rules + reusable skills + constrained agents + event-driven hooks
  4. Separating team-shared assets from personal assets is key to long-term sustainability.
  5. Context budget and permission management must be solved structurally for stable operation.
