Intro

なぜこのガイドを作ったのか

Garry Tan（Y-Combinator CEO）が公開したgstackを使ってみたところ、自分の仕事のスタイルにぴったりのハーネスだった。企画からレビュー、QA、デプロイ、振り返りまで。まるでY-Combinator時代の素早く動く初期スタートアップのような感覚を、AIハーネスで実現した感じだった。

そこで、AIコーディング環境をゼロから完全に再構築することにした。bkit、superpower、oh my codex、oh my claude codeなど数多くのハーネスをテストしてきた結果、ローカル環境には不要なファイルも多く、一度整理する必要があると感じた。

そして、どうせやり直すなら、一度整理しておけば初めての人の助けになるだろうと思い、このドキュメントを作成した。

Section 03

ハーネスエンジニアリングが
重要な理由

3-1

エンジニアリングの進化

時代コア能力核心の問い
2024プロンプトエンジニアリングAIに「何をしてほしいか」を上手く伝える技術
2025コンテキストエンジニアリングAIに「必要な情報」を上手く渡す技術
2026ハーネスエンジニアリングAIが「正しくやるしかない環境」を設計する技術

2024年の競争力：良いプロンプトを書ける人 → 2025年：エージェントを作れる人 → 2026年：AIが働く環境を設計できる人

3-2

なぜこう変化したか — LLMの発展とともに

  • 初期LLM（GPT-3〜3.5） — コンテキストウィンドウは4Kトークン。短いプロンプトの中でできるだけ明確に指示するのが全て。「プロンプトエンジニアリング」時代
  • 中期LLM（GPT-4、Claude 3） — コンテキストが128K〜200Kに拡大。コードベース、ドキュメント、履歴をまるごと入れられるように。「何を見せるか」が核心となった「コンテキストエンジニアリング」時代
  • 現在のLLM（Claude 4、GPT-5） — エージェントが自律的にコードを書き、コミットし、PRまで作成。問題は「一回の回答」ではなく「数百回の自律行動」をどう制御するか。「ハーネスエンジニアリング」時代

Karpathy: 「LLMをCPU、コンテキストウィンドウをRAMと考えよ。エンジニアの役割はOS — タスクに最適なコードとデータをメモリにロードすること。」

3-3

① コンテキストエンジニアリング — AIに何を伝えるか

新入社員に説明なしで「働け」と言っても無理です。AIも同じです。

  • プロジェクトルール、コーディング規約、作業状況… これらをCLAUDE.md、AGENTS.mdなどの文書で整理して渡す必要があります
  • エージェントが読めない情報は存在しないのと同じです
  • Cognition（Devinチーム）: 「コンテキストエンジニアリングは事実上、AIエージェントを構築するエンジニアの最優先業務だ」

Anthropic: 「長時間実行エージェントは、以前何があったか記憶なしに新しいセッションを開始する。引き継ぎ文書なしで交代勤務するエンジニアと同じだ。」

3-4

② アーキテクチャ制約 — AIがやってはいけないことを決める

ボウリングのバンパーのように、自由を与えると迷走し、境界を設けるとむしろ早く正解に辿り着きます。

  • 逆説的ですが、制約が生産性を高めます
  • Mitchell Hashimoto（HashiCorp創業者）: 「エージェントがミスをするたびに、そのミスを二度とできないようハーネスに解決策を追加せよ」
  • OpenAI Codexチーム: 依存性フロー、レイヤー分離、カスタムリンターを制約として設定し、エージェントの信頼性を劇的に向上
  • KarpathyのCLAUDE.mdがまさにこのアプローチ — 「どうコーディングするか」ではなく「やるべきでないこと」を明確にしたもの

3-5

③ ガベージコレクション — AIが作ったゴミを片付ける

AIは良いパターンも複製しますが、悪いパターンも複製します。整理しなければ品質は時間とともに低下します。

  • OpenAIチームは当初毎週金曜日を「AIスロップ清掃日」として使用 — 勤務時間の20%を整理に投入
  • 最終的に自動クリーンアップシステムをハーネスに内蔵：バックグラウンドエージェントがアーキテクチャ違反をスキャンしリファクタリングPRを自動生成
  • OpenAI: 「技術的負債は高金利ローンだ。放置すると複利で膨らみ、少しずつ返済し続ける方が常に良い」

3-6

ハーネスエンジニアリングの実際の成果

  • OpenAI Codex — 100万行以上のプロダクションコード、人間が直接書いた行数ゼロ。5ヶ月で約1,500 PR。エンジニア3人で1人あたり1日3.5 PR
  • LangChain — 同じモデル（GPT-5.2-Codex）でハーネスだけ改善したらベンチマーク52.8%→66.5%、ランキング30位圏外→Top 5
  • Stripe — 社内AIエージェントが週1,000件以上のPRをマージ

Martin Fowler: 「ハーネスエンジニアリングはコンテキストエンジニアリング、アーキテクチャ制約、ガベージコレクションを含む概念だ。」

04

4-1

問題: 両方を同じスキルで管理したい

Claude Code EnterpriseとCodex Enterpriseの両方を使用しているため、ハーネスを変更したら両方に適用する必要があります。これを簡単にするために一元管理が必要です。

  • Claude Code: /pluginでインストール → ~/.claude/plugins/cacheに保存、System Promptに自動注入
  • /pluginでインストールすると/skillsには表示されず、/contextで確認可能
  • 自動更新という利点はあるが...
  • CodexにはまだMarketplaceがない → 両方の管理問題が発生

4-2

管理方法の比較

ツール一元管理Claude/Codex同時対応判断
skills.shWebレジストリ + CLI公式対応最も完成度の高い選択
OpenSkillsGitHubベースローダー対応明記汎用パッケージマネージャ
SkillDeckローカルGUIアプリ対応含む個人Mac環境OK、チームは微妙
installagentskillsカタログ性格ガイドのみ判断補助用

4-3

最終選択: skills.sh

  • SkillDeckはMacで綺麗なGUIによる直感的な管理が可能だが、
  • ターミナル操作に慣れているのでskills.shを採用
  • npx skills check、npx skills updateで最新バージョンを維持

4-3b

なぜgstackをハーネスに選んだのか

すでにsuperpowers、bkit、omxなど色々試してきたが、gstackが自分のスタイルに一番合っていると感じたので、今回はgstackをメインハーネスとして使うことにした。gstackの詳細については後編でさらに詳しく取り上げる。

ツール強み弱み
superpowers豊富なスキルバンドル、活発なコミュニティopinionatedでないためプロセス強制力が弱い、ブラウザ自動化なし
bkit汎用性が高い、柔軟な拡張スプリントワークフロー未対応、QA/デプロイパイプライン不在
gstack企画→レビュー→QA→デプロイ→振り返りの全工程を構造化、100ms級persistent ブラウザ、並列sprint対応強くopinionated、macOS中心のクッキー復号化、iframe未対応

決定理由：プロセス強制力＋ブラウザ自動化＋品質ゲートをすべて備えた唯一の選択肢。

Section 05

gstack
詳細

05

5-1

gstackとは？

「AIコーディングエージェントをチャット型アシスタントではなく、役割が分離された仮想開発チームのように運用するためのスキルパック + 超高速ブラウザ自動化ツール」

核心は新しいモデルではなく、開発プロセスを強く構造化するという点。

GitHub: github.com/garrytan/gstack

5-2

gstackの本体

  1. SKILL.mdベースのワークフロースキル群
  2. Bun + Playwrightで構築されたpersistent browser daemon
  3. この2つをインストール/同期/検証するスクリプト群

宣伝文句を取り除くと、本質は「プロンプトOS + 高速ブラウザQA」

5-3

なぜ使うのか

プロセスの強制

企画→設計→実装→レビュー→QA→デプロイ→振り返りをコマンド型スキルで強制

高速ブラウザ

一度起動したChromiumを維持、以降のコマンド100～200ms

並列作業

ワークスペースごとに独立したブラウザ/状態/ログ

品質ゲート

コードだけでなくバグ・テスト・ドキュメント・リリースまで一括管理

5-4

インストール方法

Claude Code

git clone ...gstack.git ~/.claude/skills/gstack cd ~/.claude/skills/gstack && ./setup

Codex

git clone ...gstack.git ~/.codex/skills/gstack cd ~/.codex/skills/gstack && ./setup --host codex
  • ./setup: Bun確認 → browseビルド → Playwright Chromium → シンリンク作成 → 状態ディレクトリ準備
  • 要件: Git, Bun, ホストエージェント

5-5

実際の使用フロー

/office-hours→ /plan-ceo-review→ /plan-eng-review→ /plan-design-review
→ 実装（直接指示）→ /investigate
→ /qa→ /review→ /ship→ /document-release→ /retro

gstackは「コマンド集」ではなく、事実上スプリント運用規約である。「実装専用」スキルはなく、プランレビュー後にClaude Codeへ直接実装を指示する。

5-6

主要スキルの役割

  • /office-hours — 機能リクエストをそのまま受けず「本当に何を作りたいのか」企画をリフレーミング
  • /plan-ceo-review — プロダクトの方向性を広げたり絞ったり再設定する時
  • /plan-eng-review — アーキテクチャ、データフロー、失敗ケース、テスト計画を固める時
  • /plan-design-review — 実装前にデザインの完成度を引き上げる時
  • /design-consultation — デザインシステムをゼロから構築する時

5-7

実装後のワークフロースキル

  • /review — 構造的バグを見つける偏執的レビュー
  • /qa — 実際のブラウザでテスト実行、バグ発見時に自動修正 → 再検証
  • /ship — PR/テスト/カバレッジ/リリースの締め。テストがなければフレームワークまでブートストラップ
  • /document-release — README、CONTRIBUTINGなどのドキュメントドリフトを修正
  • /retro — 週次振り返りとチーム/個人メトリクス分析

5-8

ブラウザがなぜ重要か — /browse

  • PlaywrightをラップしたcliではなくローカルHTTP daemonにコマンドを送るthin client構造
  • 状態ファイル.gstack/browse.jsonにpid, port, tokenを記録
  • セキュリティ: localhostバインディング + bearer token + 0600権限
  • 要素選択: snapshot -iで作成した@e1, @e2 ref（アクセシビリティツリーベース）

5-9

/browse 高度な機能

  • SPAでref期限切れ → locator.count()で検知、「再度snapshotを取れ」エラーを返す
  • CAPTCHA/MFA → handoffでvisible Chrome、ユーザーが通過後resume
  • 「ログイン維持、タブ維持、Cookie維持、高速往復」— MCPブラウザツールとの核心的な違い

5-10

何が改善されるか

5-18

制限事項と注意点

  • 強くopinionated — Garry Tan式の開発文化と合わなければ窮屈に感じることがある
  • コアロジックの大部分がprompt/skill markdown → 「エンジン」より「運用ルール」の比重が大きい
  • Cookie復号化はmacOS Keychain中心（Windows/Linux未実装）
  • iframeサポートなし（most-requested missing feature）
  • 非Claudeホストでは/carefulなどがadvisory proseとして弱くなる
  • telemetryはopt-inだが存在 — スキル名、時間、結果、バージョン、OSを収集

5-19

gstack 結論

gstackは「AIがより賢くなるツール」ではなく「AIをチームのように運用する方式」である。

  • 最大の改善はモデルではなく運用構造
  • 企画再定義 → レビュー/QA/ドキュメント/リリースをスキルに分割 → ブラウザ → 並列スプリント
  • 自由度の高い汎用エージェントよりは強いプロセスと役割分離が必要な創業者/テックリード/一人開発者に適している

Section 06

CLAUDE.md

06

6-1

推奨 CLAUDE.md / AGENTS.md 構成

  • Andrej Karpathyの簡潔なCLAUDE.mdをメインとして使用
  • 4つの核心原則: Think Before Coding, Simplicity First, Surgical Changes, Goal-Driven Execution
  • Gstack指示を追加
  • サブエージェントモード指示を追加
  • RTK指示（@RTK.md参照）を追加
  • RTKは現在Claude Codeでのみ動作するため、RTK.mdを除けば~/.codex/AGENTS.mdも同一構成で問題ありません

6-1+

KarpathyのCLAUDE.mdはなぜ話題になったのか

  • たった4つの原則でAIコーディングエージェントの慢性的な問題（過剰設計、不要なリファクタリング、推測的なコード）を正面から狙い撃ち
  • 「シニアエンジニアが見て複雑すぎると言うか？ならシンプルに書き直せ」— この一文が開発者コミュニティで爆発的に拡散
  • 長く冗長だった従来のCLAUDE.mdに比べ、短く具体的な行動指針だけで実際のコード品質が目に見えて向上
  • AIに「どうコーディングするか」ではなく「やるべきでないこと」を明確にした逆転の発想が核心

6-2

Karpathyの原則 1: Think Before Coding

仮定するな、混乱を隠すな、トレードオフを明らかにせよ。

  • 仮定を明示的に示す。不確かなら質問する。
  • 解釈が複数あればすべて提示 — 黙って一つだけ選ばない
  • よりシンプルな方法があれば言う。必要なら反論する。
  • 不明確なら止まって、何が分からないか名前を付ける。

6-3

Karpathyの原則 2: Simplicity First

問題を解決する最小限のコード。推測的コードは書かない。

  • 要求範囲を超える機能なし
  • 単一使用コードに抽象化なし
  • 要求されていない「柔軟性」や「設定可能性」なし
  • ありえないシナリオに対するエラーハンドリングなし
  • 200行が50行にできるなら書き直す

6-4

Karpathyの原則 3: Surgical Changes

必要なものだけ触り、自分が作ったものだけ片付ける。

  • 隣接コード、コメント、フォーマットを「改善」しない
  • 壊れていないものをリファクタリングしない
  • 既存スタイルに合わせる（自分の好みが違っても）
  • 自分の変更で生まれた孤立コードだけ削除。既存のdead codeは言及のみ。

テスト: 変更されたすべての行がユーザーの要求に直接関連していなければならない。

6-5

Karpathyの原則 4: Goal-Driven Execution

成功基準を定義し、検証されるまで繰り返す。

  • 「バリデーション追加」→「不正入力のテストを書いてパスさせる」
  • 「バグ修正」→「再現テストを書いてパスさせる」
  • 「Xリファクタリング」→「前後でテストがパスすることを確認」
1. [ステップ] → 検証: [チェック] 2. [ステップ] → 検証: [チェック] 3. [ステップ] → 検証: [チェック]

Section 07

実践例

gstack + RalphでTodo Webアプリを作る

07

Step 1

アイデア整理 — /office-hours

Claudeがソクラテス式の質問を投げかける:

  • 「なぜこれを作りたいのですか？既存のTodoアプリと何が違いますか？」
  • 「ターゲットユーザーは誰ですか？」
  • 「最も狭いエントリーポイント（wedge）は何ですか？」

アイデア具体化 → 「開発者向けのCLI-first Todoアプリ。Webダッシュボードは閲覧用。」

Step 2

戦略レビュー — /plan-ceo-review

CEO/founder視点でプランを点検:

  • 「ダッシュボードは本当に必要か？CLIだけで十分ではないか？」
  • 「スコープを縮小してMVPをもっと早く出す方がいいのでは？」

決定: Phase 1はWeb CRUDのみ。CLIはPhase 2へ。

Step 3

エンジニアリングレビュー — /plan-eng-review

アーキテクチャを固める:

  • スタック: Next.js 15 (App Router) + Prisma + SQLite
  • モデル: Todo { id, title, completed, createdAt, updatedAt }
  • API: /api/todos (GET, POST, PUT, DELETE)
  • テスト: Vitest + Playwright E2E
  • エッジケース: 空のタイトル、重複、同時編集

Step 4

デザインシステム — /design-consultation

  • ミニマルなデザインシステムを生成
  • カラーパレット、タイポグラフィ、スペーシングルール
  • 成果物: DESIGN.md + フォント/カラープレビューページ

Step 5

実装 — Ralphループで自動化

gstackの領域を超えてRalphが登場。プランが確定したので実装をRalphループに任せる。

/ralph-loop " Phase 1: プロジェクト初期化 (Next.js + Prisma) Phase 2: API実装 (/api/todos CRUD) Phase 3: UI実装 (DESIGN.mdに従うこと) Phase 4: テスト (Vitest + Playwright, カバレッジ80%+) " --completion-promise "COMPLETE" --max-iterations 30

Step 5 (続き)

Ralphがやること

  1. Phase 1～4を順番に実装
  2. テスト実行 → 失敗したら原因分析後に修正
  3. 再テスト → 繰り返し
  4. すべてのテストがパスしたらCOMPLETEを出力 → ループ終了

人の介入は不要。夜に実行して朝に結果を確認すればよい。

Step 6–7

コードレビュー & セカンドオピニオン

/review

  • SQL安全性、信頼境界違反、条件付き副作用の構造的チェック
  • 発見された問題の修正

/codex review

  • OpenAI Codexが独立してdiffレビュー
  • pass/failゲートの役割

Step 8–9

QAテスト & デザイン監査

/qa

  • 実際のブラウザでTodoの追加/削除/完了をクリック
  • 壊れたUI、欠落したエラー処理の発見
  • バグ → 自動修正 → 再検証 → コミット

/design-review

  • ライブサイトのスクリーンショットキャプチャ
  • スペーシング、フォント、AIスロップパターンのチェック
  • 視覚的問題をソースから直接修正

Step 10–11

デプロイ & ドキュメント同期

/ship

  • ベースブランチへのマージ
  • テスト再実行（パス確認）
  • diffレビュー + VERSIONバンプ
  • CHANGELOG → コミット → プッシュ → PR

/document-release

  • READMEと実際のコードの一致確認
  • APIドキュメント、CONTRIBUTINGの更新
  • TODOSの整理

Step 12

振り返り — /retro

  • 今回のスプリントのコミット分析
  • コード品質メトリクス
  • Ralphループの繰り返し回数、どこで最も詰まったかの確認
  • 次のスプリントの改善点を導出

Appendix

Claude Code
内部構造

A

A.1

.claudeは単なる設定フォルダではない

ディレクトリ役割
commands/ワークフローのエントリーポイント
agents/隔離された役割ベースのワーカー
skills/再利用可能な知識/手順
hooks/イベント駆動の自動化
rules/コンテキスト注入の詳細ルール
settings.json権限・UI・フック・MCPの全体制御
agent-memory/エージェント永続メモリ

A.2

核心設計哲学

Commandがフローをオーケストレーションし、Agentが独立して作業し、Skillが知識/手順を再利用し、Hookが外部から自動化を掛ける。

  • 一緒に見るべきルートファイル: CLAUDE.md + .mcp.json
  • .claudeだけ見ると半分しか見えない

A.3

commands/ — ワークフローコントローラ

複雑な作業を一つのagentに詰め込まない:

  1. Commandがフローを定義
  2. Agentが独立作業を処理
  3. Skillが再利用可能な手順を担当
  • フローが見え、ユーザー確認ステップを入れやすい
  • 一つの役割が肥大化せず、再利用しやすくなる

A.4

agents/ — 役割ベースのワーカー

「fresh isolated context」を持つ独立アクター。

フィールド説明
name / description識別子 / いつ使うべきか
allowedTools / modelツール範囲 / haiku・sonnet・opus・inherit
maxTurns / skills最大ターン / preloadするスキル
memory / isolationuser・project・local / worktree

大きな万能agent一つよりも、小さな役割専用agentを複数持つ方が良い。

A.5

skills/ — 再利用可能な知識/手順

A. 直接呼び出し

commandまたはユーザーによって呼び出し。現在のコンテキストで動作。共通手順の再利用に最適。

B. Agent preload

agentのskills:に入れて開始時に注入。「このagentが知っておくべき作業知識」。user-invocable: falseでメニュー非表示が可能。

descriptionは初期コンテキストに入り、全内容は呼び出し時のみロード。長い説明はcontext budgetを消費するだけ。

A.6

hooks/ — イベント駆動の自動化

Agentic loopの中ではなく外で実行。

イベントタイミング
PreToolUse / PostToolUseツール使用の直前/直後
SessionStart / SessionEndセッション開始/終了
SubagentStart / SubagentStopサブエージェント
TaskCompleted / WorktreeCreateタスク完了 / ワークツリー作成

このリポジトリはhooks.py一つにまとめて設定ファイルでon/off — デプロイ・共有がシンプル。

A.7

rules/ & settings.json

rules/ — モジュール化されたルール

  • CLAUDE.mdにすべて詰め込まない
  • 特定ファイル群にのみ適用されるルールを分離
  • CLAUDE.mdを短く維持可能

settings.json — 全体制御

  • 権限 allow/ask/deny
  • plansDirectory、環境変数、フック
  • MCP auto-approval
  • 「利便性と安全性を両方とも構造で管理」

A.8

agent-memory/ — 動的知識の蓄積

モード範囲
memory: projectチーム共有資産
memory: userユーザー全体の記憶
memory: local個人的、gitに入らない記憶

Skill = 開始時に注入される静的な設計知識
Memory = 作業しながら蓄積される動的な運用知識
この2つを組み合わせて使うのが実践パターン。

A.9

Command vs Agent vs Skill

項目いつ使うかコンテキスト適した役割
Commandユーザーが直接開始現在のセッションエントリーポイント、順序制御
Agent独立した役割が必要新規/隔離調査、実装、レビュー
Skill再利用知識/手順現在/preload共通作業ルール

起点 → command・役割分離 → agent・再利用マニュアル → skill

A.10

分離基準 & アーキテクチャパターン

  • CLAUDE.md = 上位原則 / rules/ = 狭い範囲のルール
  • Skill = 「最初から知っていること」 / Memory = 「やりながら学んだこと」
  • settings.json = Claude動作制御 / .mcp.json = 外部サーバー接続

代表パターン: Command → Agent (with preloaded skill) → Skill
例: /weather-orchestrator → weather-agent(weather-fetcher preload) → weather-svg-creator

A.11

Best Practice (1–3)

  • 1. CLAUDE.mdは短く強く — 巨大なwiki禁止。200行以内。詳細はrules/に分離。
  • 2. 汎用より役割専用agent — 権限を絞り、memoryを分離し、prompt driftを減らす。
  • 3. Skillはprogressive disclosure — すべての知識を常に注入せず、必要な時だけ取り出す。

A.12

Best Practice (4–6)

  • 4. Hookは「事後処理」に強い — PostToolUseでフォーマッティング、Stopで検証リマインド。
  • 5. 権限はwildcardで管理 — Bash(npm run *)、Edit(/docs/**)パターン許可。destructiveはask/deny。「意図的に狭く」が基本。
  • 6. モノレポは階層的設計 — ルートCLAUDE.md（共通）+ パッケージ別CLAUDE.md（ドメイン）。一箇所に詰め込まない。

A.13

Best Practice (7–8)

  • 7. 作業は小さく分割してcontext管理 — plan modeで開始、50%付近で/compact、subtaskは50%以下で終わらせる、こまめにコミット。agent/skill/rulesを分割する理由は結局context管理。
  • 8. 人の承認ポイントはcommandに — デプロイ直前の確認、destructive actionの確認、オプション選択。agentにすべて任せると運用の統制力が落ちる。

A.14

実務向け推奨プロジェクト構造

project/ ├── CLAUDE.md # 短い上位ルール ├── .mcp.json # 外部サーバー接続 └── .claude/ ├── settings.json / settings.local.json ├── rules/ # ドメイン別詳細ルール ├── commands/ # 人が開始するワークフロー ├── agents/ # 役割分離 ├── skills/ # 再利用知識 └── hooks/ # 自動フォーマット/検証/通知

A.15

導入順序

最初からすべて導入すると過剰。段階的に:

CLAUDE.md→ settings.json→ commands→ agents→ skills→ hooks

A.16

最も実用的な結論

  1. .claudeは単なる設定フォルダではなくClaude Code運用レイヤーである。
  2. command、agent、skillは代替物ではなく役割が異なる構成要素である。
  3. 良いセットアップ = 短いメモリ + モジュール化されたrules + 再利用skills + 制限されたagents + event-driven hooks
  4. チーム共有資産と個人資産を分離しないと長続きしない。
  5. context budgetと権限管理を構造で解決しないと安定的に動作しない。
